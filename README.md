# GitHub OAuth API Testing

[![GitHub OAuth API Tests](https://github.com/haris-yousaf/OAuth-GitHub-Api-Testing/actions/workflows/newman.yml/badge.svg)](https://github.com/haris-yousaf/OAuth-GitHub-Api-Testing/actions/workflows/newman.yml)

Manual and automated API test suite for the [GitHub REST API](https://docs.github.com/en/rest) using OAuth 2.0 authorization code flow. Covers user profile, repository, and gist endpoints with a focus on authentication security, BOLA vulnerability testing, and token replay attack verification.

## What Was Tested

| Method | Endpoint | Operation |
|---|---|---|
| GET | /user | Authenticated user profile |
| GET | /users/:username | Public user profile |
| GET | /user/repos | List authenticated user repositories |
| POST | /gists | Create gist |
| GET | /gists/:id | Get gist by ID |
| PATCH | /gists/:id | Update gist |
| DELETE | /gists/:id | Delete gist |

### Test Types Covered

- Functional: valid token returns correct user data, gist CRUD operations persist correctly
- Authentication: valid token, invalid token, missing token, revoked token
- Authorization (BOLA): cross-user private gist access using a different user's token
- Security: replay attack using a manually revoked OAuth token
- Negative: non-existent usernames, deleted gist retrieval

---

## OAuth 2.0 Flow

This project uses the full authorization code flow, not a personal access token. A Node.js + Express server handles the token exchange:

```
1. Visit http://localhost:3000/login
2. GitHub redirects to authorization page
3. User approves — GitHub sends authorization code to callback URL
4. Node.js server exchanges code for access token
5. Token stored in Postman environment as githubToken
```

The `oauth-server/` folder contains the Node.js server used for this flow.

## Security Findings

| ID | Type | Endpoint | Severity |
|---|---|---|---|
| BUG-001 | BOLA (OWASP API1:2023) | GET /gists/:id | High |

**Finding:** A private gist owned by `haris-yousaf-itz` was fully accessible to `haris-yousaf` using only the gist ID and a different user's valid token. GitHub's "private" gists are unlisted, not access-controlled. Any authenticated user who knows or discovers a gist ID can read its full contents.

Full details in [`docs/bug-reports.md`](docs/bug-reports.md).

---

## Replay Attack Result

A valid OAuth token was revoked via GitHub > Settings > Applications > Authorized OAuth Apps. Subsequent requests using the revoked token returned `401 Bad credentials`, confirming GitHub invalidates tokens immediately upon revocation. No replay attack possible.

## How to Run Locally

"CLIENT_SECRET must be set as an environment variable before running the server"

**Step 1: Get an OAuth token**
```bash
cd oauth-server
npm install
node server.js
```
Visit `http://localhost:3000/login`, authorize the app, copy the token.

**Step 2: Set environment variables**
```bash
newman run postman/github-oauth-collection.json \
  -e postman/github-oauth-environment.json \
  --env-var "githubToken=YOUR_TOKEN" \
  --env-var "githubTokenUser2=YOUR_SECOND_TOKEN" \
  --env-var "revokedToken=YOUR_REVOKED_TOKEN" \
  --env-var "gistIdUser2=SECOND_ACCOUNT_GIST_ID" \
  --reporters cli,htmlextra \
  --reporter-htmlextra-export reports/newman-report.html
```

## CI/CD

All tokens are stored as GitHub Actions secrets and injected at runtime via `--env-var`. Nothing sensitive is committed to the repository.

The HTML report is generated on every Actions run and available as a downloadable artifact under the Actions tab, not committed to the repo.

**Secrets required:**

| Secret Name | Description |
|---|---|
| OAUTH_TOKEN_GITHUB | Primary account OAuth token |
| OAUTH_TOKEN_USER2_GITHUB | Second account OAuth token |
| REVOKED_TOKEN_GITHUB | Previously revoked token for replay attack test |
| GIST_ID_USER2_GITHUB | Gist ID from second account for BOLA test |
